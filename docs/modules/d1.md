# D1

## Purpose

D1 provides a Cloudflare-shaped SQLite database binding backed by WDL-owned metadata,
owner routing, and workerd localDisk storage. It is a separate stateful runtime service,
not part of user-runtime.

## Current Implementation

D1 execution lives in `d1-runtime`, a workerd service on `:8787` supervised by
`supervisor`. Loaded workers see a local facade from `runtime/d1-client.js` and
`runtime/bindings/d1.js`; the facade calls d1-runtime through Service Connect.

Control owns database lifecycle and binding resolution:

- `control/handlers/d1.js`
- `control/d1-*`
- deploy-time alias/physical-id resolution in control topology/bindings

workerd gives WDL the isolate and localDisk primitives needed to host a SQLite-backed
service, but it does not provide Cloudflare's D1 control plane, database ownership, or
routing service. WDL supplies those pieces: control metadata and tombstones, d1-runtime
owner routing, local SQLite execution, migration tracking, drain/renew, and deploy-time
freezing of a binding to a physical `databaseId`.

## Interfaces

- Tenant binding: `env.DB.prepare()`, `batch()`, `exec()`, and D1-compatible result
  shapes.
- Control lifecycle APIs: create/list/delete/migrations/execute.
- d1-runtime internal APIs for query routing, owner operations, drain/renew, and test
  hooks.
- Supervisor local endpoints: drain and renew on `127.0.0.1:8787`.
- Internal read results use `{ columns, rows }` row/column payloads between the D1
  actor, router, and runtime facade; tenant and control-facing surfaces normalize them
  back to D1-compatible object rows.
- Binary query and actor-query bodies are capped at `8 MiB` before decoding. Decoded
  requests are also capped at `1000` statements and `8 MiB` of aggregate SQL/parameter
  payload before any SQL reaches SQLite.
- `D1_MAX_RESULT_ROWS` caps returned rows per statement before the actor emits a
  response; the default is `65536` for Cloudflare D1 compatibility and overflow maps to
  `limit-exceeded`.
- `D1_MAX_RESULT_BYTES` caps aggregate returned result bytes per actor request before
  the actor emits a response; the default is `16 MiB` and overflow maps to
  `limit-exceeded`.
- Multi-statement `exec()` requests run inside one actor-local SQLite transaction. If a
  later statement fails, earlier statements in the same request are rolled back.
- D1 query/facade failures use a D1-specific payload and tenant `D1_ERROR` object rather
  than the generic platform/admin HTTP envelope. The D1 payload carries fields such as
  `success:false`, `error`, `message`, `category`, `retryable`, `statementIndex`, and
  `causeCode` as applicable.
- D1 error codes follow D1 compatibility vocabulary, including `hyphen-case` codes such
  as `limit-exceeded` and `sql-error`. Do not rewrite these to the platform
  `snake_case` vocabulary unless the D1 protocol itself changes.
- Control D1 lifecycle APIs remain admin APIs and use the control `{ error, message,
  ...details }` contract. d1-runtime internal query/classified errors are a separate
  D1 protocol surface.

## Redis / Storage Contracts

Control metadata is explicit:

- Database create starts provisional, initializes SQLite, then flips to ready.
- User-visible paths accept only ready databases.
- Delete writes tombstones and does not physically remove SQLite files.
- Deploy input may use an alias or physical id; control freezes physical `databaseId`
  and display `databaseName` into bundle metadata.

Runtime ownership:

- Ownership is per physical database, not per Service Connect target.
- Redis owner records include task identity and monotonic generation.
- Shared EFS/localDisk is safe only when the owner lease model preserves single-writer
  access.

Key families:

| Key | Type | Owner | Authority | Cleanup/delete semantics |
|---|---|---|---|---|
| `d1:databases:<ns>` | Set | Control | Namespace database index. | Membership drives namespace listing; cleanup must verify physical metadata before removing entries. |
| `d1:database:<ns>:<databaseId>` | Hash | Control | Authoritative physical database metadata and lifecycle state. | Delete moves lifecycle to tombstone rather than deleting SQLite files. |
| `d1:database-name:<ns>:<databaseName>` | String | Control | Alias/name -> physical id mapping. | Deploy WATCHes this key when resolving aliases to close recreate races. |
| `d1:database-referrers:<ns>:<databaseId>` | Set | Control | Rebuildable deployment referrer index. | Blocks database delete while workers still reference the physical id. |
| `d1:database-tombstone:<ns>:<databaseId>` | Hash | Control | Tombstone metadata for deleted physical database. | Records cleanup state; physical localDisk cleanup is out-of-band. |
| `d1:database-tombstones:<ns>` | Set | Control | Namespace tombstone index. | Used for cleanup/listing of deleted physical databases. |
| `d1:migrations-lock:<ns>:<databaseId>` | String EX | Control | Advisory migration apply lock. | Expires automatically; protects operator UX rather than storage correctness. |
| `d1:owner:db:<encoded-db-key>` | String EX | d1-runtime | Current owner record for one physical database. | Redis server `TIME` drives lease expiry; owner generation fences stale writers. |
| `d1:owner:db:<encoded-db-key>:generation` | String | d1-runtime | Monotonic owner-generation counter. | Never decremented; stale generations are rejected. |

## Ownership / Concurrency / Failure Semantics

- Any d1-runtime task may receive a request; only the owner executes SQL.
- Non-owner routers forward to the current owner or return owner hints for runtime-side
  optimization.
- Runtime owner-hint cache is an optimization only. Explicit stale-owner or ownership
  race responses clear the hint and may fall back through the router. A lost direct
  owner response is not replayed automatically, because the owner may already have
  committed a non-idempotent write; the facade returns a non-retryable
  `result-unknown` D1 error instead.
- Router read cache is best-effort freshness optimization and can be disabled with
  `D1_READ_CACHE_TTL_MS=0`.
- Drain is local-owner only. Do not call Service Connect aliases for drain or renew.

The single-writer protocol is per physical database:

1. Runtime binding metadata contains the physical `databaseId`, not just a human alias.
   Control freezes that id at deploy time and WATCHes alias and physical metadata keys
   so alias recreation races are detected.
2. d1-runtime derives the owner key from namespace and physical database id.
3. Owner claim/takeover WATCHes the owner key and generation key. A missing or expired
   owner causes the claimant to write a new owner record with TTL and a strictly higher
   generation in the same transaction.
4. SQL execution happens only on the owner. Non-owners route or return a validated owner
   hint; the actual owner still checks task id, generation, and lease freshness before
   executing. D1 does not have a separate mutable storage-pointer freshness check like
   DO because bindings are frozen to physical database ids at deploy time.
5. That owner-side check also requires at least `D1_OWNER_LEASE_GUARD_MS` of
   Redis-time lease budget remaining (default `1000`). Near-expiry owners first try a
   same-task, same-generation CAS renew; if renewal fails, they fail closed instead of
   starting a local SQLite dispatch that may overrun takeover. This narrows the
   takeover window; it is not a SQLite commit-time fence. After the Redis-time check
   succeeds, the actor converts the post-guard budget into a local monotonic deadline
   and checks that deadline before starting user SQL or pre-write schema probes in the
   synchronous dispatch path. If the deadline has passed, the next user statement does
   not start; transactional `exec`/`batch` paths roll back through `transactionSync`.
   Post-write metadata probes such as `last_insert_rowid()` do not return retryable
   ownership errors, because the user write has already committed on single-statement
   non-transactional paths. This does not interrupt a single SQLite call or result
   materialization that already started.
   D1 runtime tasks may host multiple tenants in one workerd isolate, so a pathological
   synchronous SQLite statement can still consume that isolate's CPU until the call
   returns. WDL currently treats this as an accepted shared-runtime availability
   boundary rather than enforcing a per-request SQL CPU interrupt.
6. Lease expiry is evaluated against Redis server `TIME`, not the local task clock.
   Local timers and observed-owner caches are only wake-up/routing hints.
7. The supervisor renews local owners through `127.0.0.1:8787`. Renew succeeds only if
   the Redis record still has the same task id and generation; otherwise the local task
   forgets the database.
8. Drain marks the task draining, waits for pending queries or owned actors to become
   idle, and releases only matching owner generations. Actor `wait-until-idle` probes
   wait for the d1-runtime task-local pending query count and are not themselves counted
   as SQL queries. `D1_DRAIN_TIMEOUT_MS` bounds owner-registry drain waits;
   `D1_ACTOR_IDLE_WAIT_TIMEOUT_MS` separately bounds actor-local idle probes.

This is why shared EFS/localDisk is acceptable for D1: the storage is shared, but writes
are admitted only by a Redis owner lease plus monotonic generation fence. The generation
counter survives owner record expiry so stale owners do not regain authority after a
failover.

## Security Boundaries

- D1 binding metadata is frozen at deploy time by control.
- Tenant code does not receive backend endpoints or owner credentials.
- Tenant-visible D1 metadata and errors must not include owner task ids, backend
  endpoints, or raw transport error text.
- D1 owner hints must pass service-specific endpoint grammar and generation/task checks.
- Test hooks are opt-in and must not be enabled in live-ready services.

## Observability

d1-runtime logs owner changes, routing, query failures, drain/renew, and test-hook
operations. Storage size metrics are owner-local observed state, not authoritative quota
accounting. D1-compatible query responses may still carry `meta.size_after`; the
`wdl_d1_storage_size_bytes` metric is derived only from successful samples recorded by
the current local owner and is cleared when that owner is lost or released. Forwarded
router responses must not be scraped into this gauge.

The supervisor emits `drain_timeout_near_owner_ttl` when configured drain timeout is at
least half the owner TTL. The warning carries `drain_timeout_ms`, `owner_ttl_ms`, and
`renew_interval_ms` so operators can see whether shutdown drain may run too close to
lease expiry.

## Deployment / Rollout Notes

- Roll d1-runtime with user/system runtime when D1 transport or facade protocol changes.
- Roll control with runtime when D1 metadata shape changes.
- Drain d1-runtime locally before stopping workerd to avoid owner handoff races.
- EFS/localDisk behavior needs operational care during host changes.
- If drain or owner TTL environment variables change, keep supervisor termination grace
  longer than the configured drain window plus margin.
- `D1_TEST_HOOKS=1` exposes internal test hooks that can hold SQLite transactions for
  crash-recovery tests. Use it only in disposable integration runs; compose tests
  that need it must recreate D1 runtime containers because env substitution only applies
  at container creation.

## Known Constraints And Non-Goals

- WDL does not implement Cloudflare global replication or bookmark semantics.
- Physical SQLite files are not deleted on metadata delete.
- `wrangler d1` management commands are not the platform management surface.
- Router-side read cache is not a correctness proof.
- D1 does not transparently replay a direct owner request after the owner response is
  lost. Callers must decide whether a `result-unknown` failure is safe to retry for
  their workload.
- Storage size metrics are owner-local observed state, not quota accounting.
