# D1

## 目的

D1 提供 Cloudflare-shaped SQLite database binding，但底层由 WDL 自己的 metadata、owner routing 和 workerd localDisk storage 支撑。它是独立 stateful runtime service，不属于 user-runtime。

## 当前实现

D1 执行在 `d1-runtime` 中，这是监听 `:8787`、由 `supervisor` 管理的 workerd service。Loaded worker 看到来自 `runtime/d1-client.js` 和 `runtime/bindings/d1.js` 的本地 facade；facade 通过 Service Connect 调用 d1-runtime。

Control 拥有 database lifecycle 和 binding resolution：

- `control/handlers/d1.js`
- `control/d1-*`
- control topology/bindings 中的 deploy-time alias/physical-id resolution

workerd 给 WDL 提供了托管 SQLite-backed service 所需的 isolate 和 localDisk primitives，但不提供 Cloudflare D1 的 control plane、database ownership 或 routing service。WDL 补齐这些部分：control metadata 和 tombstone、d1-runtime owner routing、本地 SQLite execution、migration tracking、drain/renew，以及 deploy 时把 binding 冻结到 physical `databaseId`。

## 接口

- Tenant binding：`env.DB.prepare()`、`batch()`、`exec()` 和 D1-compatible result shape。
- Control lifecycle APIs：create/list/delete/migrations/execute。
- d1-runtime 内部 API：query routing、owner operations、drain/renew、test hooks。
- Supervisor local endpoints：`127.0.0.1:8787` 上的 drain 和 renew。
- D1 actor、router 和 runtime facade 之间的内部 read result 使用 `{ columns, rows }` row/column payload；tenant 和 control-facing surface 会再 normalize 回 D1-compatible object rows。
- Binary query 和 actor-query body 会在 decode 前限制为 `8 MiB`。Decoded request 在任何 SQL 进入 SQLite 前还会限制为最多 `1000` 条 statement，以及 `8 MiB` aggregate SQL/parameter payload。
- `D1_MAX_RESULT_ROWS` 限制单条 statement 返回的最大行数，actor 返回响应前执行检查；默认值为 `65536` 以贴近 Cloudflare D1 兼容性，超限映射为 `limit-exceeded`。
- `D1_MAX_RESULT_BYTES` 限制单个 actor request 返回的 aggregate result bytes，actor 返回响应前执行检查；默认值为 `16 MiB`，超限映射为 `limit-exceeded`。
- Multi-statement `exec()` request 会在一个 actor-local SQLite transaction 中执行。如果后续 statement 失败，同一 request 中此前已执行的 statement 会回滚。
- D1 query/facade 失败使用 D1-specific payload 和 tenant `D1_ERROR` object，而不是 generic platform/admin HTTP envelope。D1 payload 会按需携带 `success:false`、`error`、`message`、`category`、`retryable`、`statementIndex` 和 `causeCode` 等字段。
- D1 error code 使用 D1 compatibility vocabulary，包括 `limit-exceeded`、`sql-error` 这类 `hyphen-case` code。除非 D1 protocol 本身变化，否则不要把它们改写成平台 `snake_case` vocabulary。
- Control D1 lifecycle API 是 admin API，使用 control `{ error, message, ...details }` 合同。d1-runtime internal query/classified error 是独立的 D1 protocol surface。Control 对外状态为 5xx 时会把 backend message 替换为 `Internal error`、移除 raw detail，只保留 `upstreamCode`、`upstreamCategory`、`upstreamRetryable` 和 `upstreamStatus` 有界机器分类字段；对外 4xx SQL 与 migration failure 会暴露 D1 diagnostic detail。

## Redis / Storage 合同

Control metadata 是显式 lifecycle：

- Database create 先写 provisional，初始化 SQLite，再 flip 到 ready。
- 用户可见路径只接受 ready database。
- Delete 写 tombstone，不物理删除 SQLite 文件。
- Deploy input 可使用 alias 或 physical id；control 将 physical `databaseId` 和显示用 `databaseName` 冻结进 bundle metadata。
- Runtime materialization、query decoding 和 rebalance 使用共享 canonical grammar 重新校验 namespace 与 physical `databaseId`。

Runtime ownership：

- Ownership 按 physical database 划分，不按 Service Connect target。
- Redis owner record 包含 task identity 和 monotonic generation。
- Shared EFS/localDisk 只有在 owner lease 模型保证 single-writer 时才安全。

workerd 2026-07-01 会大小写不敏感地拒绝 SQLite reserved `_cf_` namespace 下的 object name，包括 `ALTER TABLE ... RENAME TO _cf_*`。如果既有 database 中已经有 `_CF_*` 等大小写变体，应把它们视为 reserved upgrade debris，而不是应用拥有的表。

Key families：

| Key | Type | Owner | Authority | Cleanup/delete 语义 |
|---|---|---|---|---|
| `d1:databases:<ns>` | Set | Control | namespace database index。 | membership 驱动 namespace list；cleanup 移除 entry 前必须验证 physical metadata。 |
| `d1:database:<ns>:<databaseId>` | Hash | Control | physical database metadata 和 lifecycle state 的权威记录。 | Delete 转成 tombstone lifecycle，不删除 SQLite 文件。 |
| `d1:database-name:<ns>:<databaseName>` | String | Control | alias/name -> physical id 映射。 | Deploy 解析 alias 时 WATCH 这个 key，关闭 recreate race。 |
| `d1:database-referrers:<ns>:<databaseId>` | Set | Control | 可重建 deployment referrer index。 | worker 仍引用 physical id 时阻止 database delete。 |
| `d1:database-tombstone:<ns>:<databaseId>` | Hash | Control | deleted physical database 的 tombstone metadata。 | 记录 cleanup state；physical localDisk cleanup 是异步/外部过程。 |
| `d1:database-tombstones:<ns>` | Set | Control | namespace tombstone index。 | 用于 deleted physical database 的 cleanup/listing。 |
| `d1:migrations-lock:<ns>:<databaseId>` | String EX | Control | advisory migration apply lock。 | 自动过期；保护 operator UX，不是 storage correctness fence。 |
| `d1:owner:db:<encoded-db-key>` | String EX | d1-runtime | 单个 physical database 的当前 owner record。 | Lease expiry 以 Redis server `TIME` 为准；owner generation fence stale writer。 |
| `d1:owner:db:<encoded-db-key>:generation` | String | d1-runtime | monotonic owner-generation counter。 | 不递减；stale generation 会被拒绝。 |

## Ownership / 并发 / 失败语义

- 任意 d1-runtime task 都可收到请求；只有 owner 执行 SQL。
- 非 owner router 转发到当前 owner，或返回 owner hint 供 runtime 优化。
- Runtime owner-hint cache 只是优化；明确的 stale-owner 或 ownership race response 会清除 hint，并且可以回退到 router。Direct owner response 丢失时不会自动重放，因为 owner 可能已经提交了非幂等写入；facade 会返回不可重试的 `result-unknown` D1 error。
- Router read cache 是 best-effort freshness 优化，可用 `D1_READ_CACHE_TTL_MS=0` 关闭。
- Drain 是 local-owner only。不要通过 Service Connect alias 调用 drain 或 renew。

单写入协议按 physical database 划分：

1. Runtime binding metadata 保存 physical `databaseId`，不只保存人类可读 alias。Control 在 deploy 时冻结这个 id，并 WATCH alias key 和 physical metadata key，所以 alias recreate race 会被检测到。
2. d1-runtime 从 namespace 和 physical database id 派生 owner key。
3. Owner claim/takeover 会 WATCH owner key 和 generation key。owner 缺失或过期时，claimant 在同一个 transaction 中写入带 TTL 的新 owner record，并使用严格递增的 generation。
4. SQL 只在 owner 上执行。非 owner 负责 route 或返回已校验的 owner hint；真正的 owner 执行前仍检查 task id、generation 和 lease freshness。D1 不像 DO 那样还有单独的 mutable storage-pointer freshness check，因为 binding 在 deploy time 已冻结到 physical database id。
5. 该 owner-side check 还要求 Redis-time lease 剩余至少 `D1_OWNER_LEASE_GUARD_MS`（默认 `1000`）毫秒。接近过期的 owner 会先尝试 same-task、same-generation CAS renew；如果 renew 失败，才 fail closed，而不是开始一次可能跨过 takeover 的本地 SQLite dispatch。它缩窄 takeover window，但不是 SQLite commit-time fence。Redis-time check 成功后，actor 会把扣除 guard 后的预算转换成本地 monotonic deadline，并在同步 dispatch path 中开始用户 SQL 或写前 schema probe 前检查该 deadline。deadline 已过时，下一条用户 statement 不会开始；事务型 `exec`/`batch` 路径通过 `transactionSync` 回滚。`last_insert_rowid()` 这类写后 metadata probe 不会返回 retryable ownership error，因为单条非事务路径上的用户写入已经提交。它不会中断已经开始的单个 SQLite call 或结果物化。D1 runtime task 可能在同一个 workerd isolate 中承载多个 tenant，因此病态同步 SQLite statement 仍可能在该调用返回前消耗这个 isolate 的 CPU。WDL 当前把它作为 accepted shared-runtime availability boundary，而不是实现 per-request SQL CPU interrupt。Terraform 除了 Fargate task memory limit，还会给 d1-runtime workerd container 设置显式 memory hard limit；这是 container failure boundary，不是 per-request SQL memory interrupt。
6. Lease expiry 以 Redis server `TIME` 为准，不使用本地 task clock；本地 timer 和 observed-owner cache 只是 wake-up / routing hint。
7. Supervisor 通过 `127.0.0.1:8787` renew 本地 owner。只有 Redis record 仍是相同 task id 和 generation 时 renew 才成功；否则本地 task forget 该 database。
8. Drain 会把 task 标成 draining，等待 pending query 或 owned actor idle，并且只释放匹配 generation 的 owner。Actor `wait-until-idle` probe 会等待 d1-runtime task-local pending query count，probe 本身不会被计入 SQL query。`D1_DRAIN_TIMEOUT_MS` 只约束 owner-registry drain wait；`D1_ACTOR_IDLE_WAIT_TIMEOUT_MS` 单独约束 actor-local idle probe。

这就是 D1 可以使用 shared EFS/localDisk 的原因：storage 是共享的，但写入只由 Redis owner lease + monotonic generation fence 放行。Generation counter 在 owner record 过期后仍保留，因此 stale owner 不会在 failover 后重新获得写权限。

## 安全边界

- D1 binding metadata 由 control 在 deploy time 冻结。
- Tenant code 不获得 backend endpoint 或 owner credential。
- Tenant-visible D1 metadata 和 error 不得包含 owner task id、backend endpoint 或原始 transport error 文本。
- D1 owner hint 必须通过 service-specific endpoint grammar，并携带 task id 和正的 JavaScript-safe-integer generation。
- Task identity、rebalance target 和 persisted owner record 在写入和读取时都会校验。Persisted record 必须能重建出读取它的 physical database key；错放在其他 database scope 下的记录会 fail closed。Owner forwarding 只接受 8787 端口上的 D1 service/headless DNS，或 RFC1918/100.64 私网 IPv4；非法记录在附加 internal auth 前 fail closed。
- Test hooks 是 opt-in，不应在 live-ready service 中开启。

## 可观测性

d1-runtime 记录 owner 变化、routing、query failure、drain/renew 和 test-hook 操作。Storage size metrics 是 owner-local observed state，不是权威 quota accounting。D1-compatible query response 仍可携带 `meta.size_after`；`wdl_d1_storage_size_bytes` 只来自当前 local owner 成功记录的 sample，并在 owner 丢失或释放时清除。不要把 forwarded router response scrape 进这个 gauge。

Control 的 database initialization、query execution 和 migration operation runtime failure event 会包含 request/database identity，以及 `upstream_status`、`upstream_code`、`upstream_category`、`upstream_retryable` 和可用时的 `upstream_cause_code` machine fields；Control 不会把原始 backend message 或 response detail 复制进这些 event。

当配置的 drain timeout 大于等于 owner TTL 的一半时，supervisor 会输出 `drain_timeout_near_owner_ttl`。该 warning 携带 `drain_timeout_ms`、`owner_ttl_ms` 和 `renew_interval_ms`，方便 operator 判断 shutdown drain 是否过于接近 lease expiry。

## 部署 / Rollout 注意事项

- D1 transport 或 facade protocol 改变时，d1-runtime 应与 user/system runtime 一起滚。
- D1 metadata shape 改变时，control 和 runtime 应一起滚。
- 停止 workerd 前应本地 drain d1-runtime，避免 owner handoff race。
- EFS/localDisk 在 host 变化时需要运维关注。
- 如果修改 drain 或 owner TTL 环境变量，supervisor termination grace 必须长于配置的 drain window，并留出余量。
- `D1_TEST_HOOKS=1` 会暴露可持有 SQLite transaction 的内部 test-hook endpoint，只能用于一次性 integration 运行，不能用于普通 local dev 或 live-ready service。需要这些 hook 的 compose 测试必须 recreate D1 runtime container，因为 env substitution 只在 container 创建时生效。

## 已知约束和非目标

- WDL 不实现 Cloudflare 全球复制或 bookmark 语义。
- Metadata delete 不物理删除 SQLite 文件。
- `wrangler d1` 管理命令不是平台管理面。
- Router-side read cache 不是正确性证明。
- D1 不会在 direct owner response 丢失后透明重放请求。调用方必须根据自身 workload 判断 `result-unknown` 是否可以重试。
- Storage size metric 是 owner-local observed state，不是 quota accounting。
